This is a new feature not yet documented The safest and best way to operate robopages is in a bash/scp only mode. The site admin can chown -R all files to a user who is not the apache process and chmod -R 555 the fragments directory, chmod or chown any LOGS directories, in order make them writable to the web server process.. Then it's secure.

You might want to build a robopages site for a customer however, who doesn't do bash perl python sed awk and scp.

In that case the RobopageAdmin.php interface is a work in progress. I'm not making any promises as to it's security as of today. I do have one user using it.

Once I get it cleaned up enough I'll spend some time working on security issues. The various action="?layout=nerd" method="post" forms need to make more use of md5 tokens,etc. Right now it works and it's safe because only one site in the universe uses it. And no one else knows. Better security will happen in time

To make this work you have to chown -R apache.apache the fragments directory. You may have to chmod -R 555 too if it's on a virtual dedicated server.

The admin interface does a recursive chmod -R 755 to make it writable during the admin session. If the user remembers to click the Logout button it chmods back to 555 when done.

Authentication is files based. Look for RobopagePasswords.php in the plugins directory. the RobopagePasswords.php file has the form usename:md5password:nimda, where nimda (admin backwards) is hard-coded.

How to do it

got to the conf directory. copy RobopageAdminPlugins.ini onto plugins.ini edit the plugins/RobopagePasswds.php file appropriately. There is no "login" link in the codes. You have point a browser at the document root and then manually add ?layout=nerd to see a login screen. Click the logout button as you leave.
